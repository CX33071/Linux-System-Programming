1.概念
信号是一种软件中断，用于通知进程发生了某个异步事件，用于进程间通信
信号来源：
    硬件异常：除0错误、非法访问内存
    终端按键：ctrl+c(SIGINT),ctrl+\(SIGQUIT)
    系统调用：kill(),raise()
    内核事件：子进程退出SIGCHLD、定时器到期SIGALRM
信号生命周期：产生，内核推送，进程处理(执行默认动作、忽略、捕获)
信号编号通常从1开始，信号是异步的，进程无法预知信号何时到达，因此信号处理函数必须是可重入的
2.信号类型和默认行为
SIGINT(2)：中断信号，通常由ctrl+c触发，默认终止进程
SIGTERM(15):终止信号，kill命令的默认信号，用于优雅地终止进程
SIGKILL(9)：强制终止进程，无法被捕获、阻塞或忽略，是终结失控进程的最后手段
SIGSTOP(19/17/23):停止信号，无法被捕获、阻塞或忽略，暂停进程执行
SIGCHLD(17/18/20):子进程状态改变退出暂停恢复时，父进程会收到此信号
SIGSEGV(11):段错误，进程访问非法内存时触发
默认行为：
终止Terminate:进程直接退出
终止并转储核心：进程退出并生成核心转储文件，用于调试
忽略：内核直接丢弃信号，进程不受影响
停止：暂停进程执行
继续：恢复被暂停的进程
SIGKILL和SIGSTOP的行为是内核级别的，不能被用户空间代码改变
3.注册信号处理函数signal()
#include <signal.h>
void (*signal(int sig,void(*func)(int)))(int)
返回值本质是一个函数指针，调用成功返回该信号之前的处理函数指针，即信号上一次设置的1处理方式，常见的返回值包括：SIG_DFL默认处理
        SIG_IGN忽略该信号
        SIG_ERR调用失败
参数sig：要处理的信号编号
参数handle：信号处理函数指针SIG_DFL恢复该信号的默认行为   SIG_IGN忽略该信号
缺点：不支持信号掩码，无法在处理函数中阻塞信号
4.信号处理器
- 信号处理函数：当信号递送给进程时内核会暂停进程的正常执行，跳转到注册的处理函数的执行，执行完毕再恢复
- 可重入性：信号处理函数可能在进程执行任何代码时被调用
            仅调用异步信号安全函数write(),_exit()等，避免使用printf(),malloc()
            write和_exit是内核态系统调用，内核通过自身的同步机制保证安全，而printf/malloc/exit是用户态库函数，为了保证多线程安全，内部封装了用户态互斥锁
            write()是内核实现的系统调用，操作内核缓冲区且保证原子性，无用户态全局数据修改，不会因为信号异步执行产生竟态，而printf依赖全局静态缓冲区信号中断时会导致缓冲区数据错乱
            _exit()是exit()的内核版本，仅内核态终止进程，无用户态副作用，不依赖非安全的全局状态或锁，而exit()可能因刷新stdio缓冲区调用fflush()导致死锁
        互斥锁：保证同一时间只有一个进程或者线程能访问某个共享资源避免竟态，同一时间只能有一个线程持有锁，其他线程尝试加锁会被阻塞，加锁/解锁是原子的不会被中断，被阻塞的县城进入休眠状态直到锁被释放，死锁问题指多个线程互相等待对方持有的锁导致所有线程永远阻塞。信号处理函数中禁止使用互斥锁，如果主程序持有锁，信号出来函数被触发并尝试加锁，此时主程序被打断不能解锁可信号处理函数在等锁此时导致死锁
            信号处理函数访问全局变量时必须将全局变量声明为volatile sig_atomic_t类型，因为1编译器优化认为变量不会被修改，可能会把全局变量缓存到寄存器中，主程序一直读取寄存器的旧的值完全看不到信号处理函数修改的内存中的新值1,主程序和信号处理函数看到的变量值不一致2非原子操作，信号中断可能发生在操作中途导致数据错乱，volatile告诉编译器不要对这个变量任何优化，每次访问都必须直接读写内存。sig_atomic_t保证操作是原子性的，避免在32位系统中先写低32位再写高32位中间被信号处理函数打断
        pause():让当前进程暂停执行直到收到一个非忽略的信号进程才会被唤醒
5.kill()发送信号
kill()用于向指定进程或进程组发送信号
#include <signal.h>
int kill(pid_t pid,int sig);
参数pid:    pid>0   向进程ID为pid的进程发送信号
            pid=0   向与发送进程同组的所有进程发送信号
            pid=-1  向所有有权限发送信号的进程发送，除了init进程
            pid<-1  向进程组ID为-pid的所有进程发送信号
权限：发送者必须是目标进程的所有者，或root
6.检查进程的存在
像一个进程发送信号0(空信号)，可以检查该进程是否存在，若kill(pid,0)返回0说明进程存在，若返回-1且errno为ESRCH，说明进程不存在
7.发送信号的其他方式raise()/killpg()
raise()向自身进程发送一个信号，等价于kill(getpid(),sig)
killpg()向指定进程组发送信号，等价于kill(-pgrp,sig)
8.显示信号描述
信号描述函数:strsignal()和psignal()用于获取和打印信号的描述性字符串，比单纯的信号编号更易读，避免硬编码信号含义
strsignal()返回指向信号描述字符串的指针，char*strsignal(int sig)
void psignal(int sig,const char*s);在标准错误输出信号描述，可附带自定义信息
9.信号集
信号集sigset_t sigset类似一个数组，数组元素是信号，表示一组信号
信号集操作函数：sigemptyset()   清空信号集
             sigfillset()   将所有信号加入信号集
             sigaddset()    向信号集中添加一个信号
             sigdelset()    从信号集删除一个信号
             sigismember()  检查一个信号是否在信号集中
10.信号掩码
每个进程都有一个信号掩码，它是一个信号集，指定了当前被阻塞的信号，被阻塞的信号处于pending状态，不会递送给进程，直到被解除阻塞
修改信号掩码：sigprocmask()
#include <signal.h> int sigprocmask(int how,const sigset_t *set,sigset_t *oldset);
参数how:    SIG_BLOCK   将set中的信号添加到当前掩码
            SIG_UNBLOCK     从当前掩码中移除set中的信号
            SIG_SETMASK     将当前掩码设置为set
11.处于等待状态的信号
pending信号：被阻塞无法递送的信号
sigpending(sigset_t *set)函数返回当前进程的pending信号集
12.信号的排队处理
标准信号1-31：不支持排队，如果一个信号已经处于pending状态，此时又产生同一个信号，那么后产生的信号被丢弃，进程只收到一次该信号
实时信号32-64：支持排队
13.sigaction()
比signal强大可靠，庚推荐
int sigaction(int sig,const struct sigaction*act,struct sigaction *oldact);
struct sigaction 结构体：
struct sigaction {
    void (*sa_handle);信号处理函数
    void (*sa_sigaction);替代处理函数
    sigset_t sa_mask;处理信号时要阻塞的其他信号
    int sa_flags;控制行为的标志
    void *sa_restorer;已废弃
}
sa_flags:SA_RESTART让被信号中断的系统调用自动重启
         SA_NOCLDSTOP当子进程停止时不产生SIGCHLD
14.等待信号pause()
使进程暂停执行，直到收到一个信号，信号处理函数执行完毕后，pause()返回-1,并将errno设置为EINTR
在等待信号时让进程进入休眠状态，避免占用CPU
